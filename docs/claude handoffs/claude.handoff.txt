THE CRAFTING RIP-OUT (completed this session)
-------------------------------------------------------------
WHY: the spec is a complete redesign. Sampling was deleted, quality
rescales, recipes become blueprints, items need craft_seed, materials
need provenance, experimentation/sockets/durability/workshops did not
exist. Refactoring toward it would have meant new code sitting beside
dead code -- the exact half-state that caused earlier bugs.

STEP 1 - THE SEAM (done first, non-destructive):
  `GameData.recipes` was doing DOUBLE DUTY: crafting recipe AND the
  only weapon catalog in the game. Deleting it would have deleted every
  weapon. So item identity was split out first:

    GameData.ITEM_DEFINITIONS   what an item IS (item_class,
                                item_subclass, weapon_categorical_stats,
                                weapon_stat_ranges, max_charges)
    GameData.get_item_definition(name)

  Combat, equipment and inventory read ITEM_DEFINITIONS and never touch
  recipes. Only ONE surviving function depended on recipes
  (_grant_starting_weapon) plus the validator; both repointed.

STEP 2 - THE CUT:
  main.gd     5760 -> 4371 lines, 193 -> 152 functions
  GameData.gd 1053 -> 685 lines

  41 functions deleted: crafting book, recipe tree, assembly flow,
  result popup, survey book, hotspot generation, sampling, plus
  orphaned helpers (_matches_requirement, _get_weighted_stack_score,
  _get_unlocked_classes, _get_ingredient_breakdown_text, and others).
  39 declarations deleted. Survey state stripped from save/load.
  The `recipes` array is gone entirely.

  KEPT: all 13 inventory functions, _generate_unique_resource_name,
  and _attempt_forage (the dumpster) -- deliberately, as the seed of
  the new scavenging model.

  ITEM_DEFINITIONS retained 23 entries: 19 weapons + 4 medicine.
  Deleted 8: 5 Tools (survey tools, crafting kits), 3 Components
  (Syringe, Empty IV Bag, Weapon Muzzle -- the last a test item).

CONSEQUENCES TO KNOW:
  - MEDICINE IS NOW UNOBTAINABLE beyond the starting grant. Bandages,
    Salve, Tonic and Adrenaline Shot still exist and still work, but
    nothing produces more. Finite supply per save.
  - CRAFTING XP HAS NO SINK. It still accrues; nothing spends it.
    Deliberately deferred -- the game is not in a playable state yet.
  - Loot tables still drop Copper/Iron etc. Those items are now inert.
    Data debt, cheap to fix once the new material model is settled.

NEW CRAFTING SYSTEM (rebuild in progress)
-------------------------------------------------------------
LOCATION: C:\My Game\Steamtek-RPG\systems\crafting\
No autoload registration needed -- all use `class_name` with static
functions, reachable as CraftingData.X etc from anywhere.

FILES:
  crafting_data.gd               CraftingData
      static definitions only. 22 resource families, 6 traits,
      5 instabilities, 6 quality tiers, 2 starter blueprints,
      Silo structure (60 floors / 3 eras), 15 floor identities,
      resource guarantees.
  crafting_models.gd             CraftingModels
      the three runtime shapes + factories + to_dict/from_dict:
      MaterialBatch, CraftedItemInstance, PlayerCraftingProfile.
  crafting_service.gd            CraftingService
      the only file that CHANGES state: batch generation, material
      potential, validation, craft(), keystone 4-node cap, save/load.
  crafting_resource_generator.gd CraftingResourceGenerator
      seeded campaign map generation, guarantee enforcement,
      extract_from_source(), save/load.

DESIGN DECISIONS MADE:
  - Plain Dictionaries, NOT Resource/.tres. The save file is JSON so
    Dictionaries round-trip with no conversion layer; no editor assets
    to hand-manage; avoids the shared-mutable-state footgun the spec
    itself warns about for per-item instances.
  - Material Quality is 0-100 (spec). This is a DIFFERENT scale from
    the legacy weapon "Quality" (0-1000) still used by equipment.
    Do not mix them.
  - 4-of-15 keystone selection applies to CRAFTING ONLY. Combat
    keystones keep the point-budget model. (Spec s2.2 implies 4-of-15
    for every class keystone; Josh scoped it to crafting, which avoids
    a full keystone-system and KeystoneViewer rewrite.)
  - The crafting keystone gets its OWN UI/panel, separate from the
    talent viewer. Current UI is placeholder throughout.
  - Every crafting keystone node needs a DOWNSIDE. As written in the
    spec, Material Insight and Precision Calibration are pure upside
    and would be auto-picks, violating the spec's own "avoid four
    universally optimal picks" rule.

PHASE 1 - DATA FOUNDATION (DONE)
  Acceptance met: one blueprint + three material batches produce a
  saved crafted item instance; reload preserves all values (verified
  by simulation).
  craft_seed is stored on the item FROM DAY ONE -- later phases must
  derive randomness from it, never roll fresh, or reload-scumming
  becomes possible.
  Provenance (source_id / floor_id / location_id) is on every batch.

PHASE 2 - CAMPAIGN RESOURCE GENERATOR (DONE)
  60 floors: Survivors 1-20 (quality bias -8), Expansion 21-40 (0),
  Builders 41-60 (+12). Deeper is OLDER but cleaner and better made.
  15 floor identities distributed by era. 22 resource families.
  All four acceptance criteria verified:
    same seed -> identical map; different seeds -> different valid
    maps; reload never rerolls (generation runs once at new game);
    guarantees always satisfiable.
  extract_from_source() is the seam the new scavenging loop plugs into.

  TUNING CAVEAT: guarantees are satisfied naturally on ~33% of seeds;
  the other ~67% get deterministic repair. Repairs produce valid maps,
  but firing that often means the safety net is doing more work than
  intended. An earlier pass was far worse (90% repaired) because the
  guarantee thresholds contradicted the quality curve -- Survivors-era
  common sources cap near 72 but the rule demanded 80+. Thresholds
  were retuned and gunmetal_steel widened from 2 to 4 floor
  identities. These are placeholder numbers (spec s24); real
  calibration needs playtesting.

REMAINING SPEC PHASES:
  4 experimentation
  5 mod sockets
  6 mods
  7 crafting keystone (the 15-node, choose-4 tree)
  8 repair / dismantle / rebuild

PHASE 3 - BASIC CRAFTING (DONE, wired into the game)
-------------------------------------------------------------
Built in three stages deliberately, because main.gd is large and
function-span deletion has bitten twice (see INCIDENTS).

3a - DATA (standalone files, zero risk to main.gd)
  Two MEDICINE blueprints added: bp_bandages, bp_antiseptic_salve.
  Both max_sockets 0 -- consumables take no mods. Bandages are the
  cheapest recipe in the game on purpose: medicine had become
  unobtainable when the old system was cut, and this ends that.
  SURFACE SALVAGE added. The Surface sits outside the Silo's 60
  generated floors, so it gets its own synthetic source set:
    Alley Dumpster   scrap_plating (40), cultivated_fiber (30),
                     insulating_polymer (20)
    Stripped Wiring  copper (24)
    Rusted Railing   black_iron (24)
  Quality band 12-34 of 100 -- street salvage is picked over BY
  DESIGN. Surface-crafted gear is poor; real quality means descending.
  Bandages are craftable from surface materials alone, so a player can
  craft before ever entering the Silo.
  generate_surface_sources() is seeded like everything else and never
  touches the Silo map.

3b - MAIN.GD INTEGRATION
  New state: campaign_seed, campaign_map, surface_sources,
  material_batches, crafted_items, crafting_profile.
  New functions: _init_crafting_campaign, _store_material_batch,
  _batches_for_slot, _consume_selection, _grant_crafted_item,
  _perform_craft, _load_crafting_state.
  THE DUMPSTER IS NOW A REAL SCAVENGING SOURCE. It pulls from the
  Surface sources via extract_from_source(), yielding a MaterialBatch
  with quality, trait, provenance and a depleting `remaining`. The flat
  Plastic drop is gone. This is the same call the Silo scavenging loop
  will use.
  Batches of identical family/quality/trait/instability/source MERGE so
  repeated scavenging does not flood the store; anything differing stays
  separate, because quality and provenance are meaningful.
  Campaign map generates ONCE on new game and is saved WHOLE. A load
  restores it verbatim and never regenerates -- the anti-reroll rule.

3c - CRAFTING PANEL (scenes/CraftingPanel.gd)
  Built in code, mirroring KeystoneViewer (extends Node, setup(parent)).
  Blueprint list, per-slot material dropdowns showing name/quality/held,
  live material-potential preview with quality tier, craft button that
  disables until the selection validates. Viewport-anchored from the
  start; closes via X or ESC.
  REQUIRES an input action named "crafting_menu" (Project Settings ->
  Input Map). Guarded by InputMap.has_action() so a missing action
  cannot crash the game -- the panel simply will not open.

QUALITY SCALE BRIDGE -- THE ONE PLACE THE TWO SCALES MEET:
  crafting quality is 0-100; the legacy weapon model realises stats as
  min + (Quality / 1000) * (max - min). _grant_crafted_item() scales the
  crafted score x10 into inventory_stats["Quality"]. Without that every
  crafted weapon would sit at the bottom of its range. Do not "simplify"
  this away.

KNOWN LIMITS (expected, not bugs):
  - Crafted quality is material potential passed straight through, so
    the same materials always give the same result. Phase 4
    (experimentation) is what makes allocation a decision.
  - Loot tables still drop Copper/Iron etc. as inert plain items. They
    are NOT MaterialBatches. Data debt, cheap to fix.
  - Crafting XP still has no sink.

INCIDENTS AND LESSONS
-------------------------------------------------------------
1. GAMEDATA DESYNC. GameData.gd was stale on BOTH machines through
   combat Phases 4c-7, carrying pre-4b damage type names. Twelve
   weapons matched no resistance key, got rating 0, and bypassed enemy
   armor entirely -- the archetype resistance system was inert in play
   the whole time it was being built on. Failed silently because a
   missing key returns a legal zero.
   LESSON: install delivered files on both machines the same day. The
   startup validation guard now catches this class of bug, but only
   AFTER the fact.

2. FUNCTION-SPAN DELETION (happened TWICE, once in combat 6b and again
   during the crafting cut). A bulk edit defined a function's end as
   the NEXT `func` keyword. Any top-level declaration block sitting
   between two functions was swallowed. First time: ~725 lines and 30+
   unrelated functions. Second time: 80 declarations including
   BASE_HIT_CHANCE, MELEE_RANGE, inventory, equipped_weapon_name.
   Both recovered from a pre-edit backup with no loss.
   LESSON: a function ends at the first COLUMN-0 non-blank line, not
   at the next func. Take a backup before any bulk edit. Compare
   DECLARATION COUNT before and after -- that is the check that
   catches this immediately. Dry-run large cuts before executing.

3. LOST DECLARATION AFTER RECOVERY. Restoring from a snapshot rolled
   back a `var enemy_nodes` declaration while the redo rebuilt only
   the functions USING it -- 9 "not declared in current scope" errors.
   LESSON: after recovering from a snapshot, re-check that every
   declaration the refactor introduced still exists.

4. RIGHT_X ORPHAN. A refactor REMOVED a declaration while two child
   controls still referenced it.
   LESSON: the mirror of #2. Grep for users before deleting a
   declaration. The undeclared-identifier sweep now runs before every
   export.
5. GUESSED FIELD NAMES ON THE CRAFTING MODELS (three times).
   main.gd was written against assumed key names: item_id (really
   instance_id), stats (really final_stats), generated_name (really
   display_name). Each failed SILENTLY because .get(key, default)
   returns the default for a wrong key -- the same shape as the
   GameData desync and the quality-50 drag.
   LESSON: read the model factory before reading its dictionaries. An
   audit now runs before export, checking every key main.gd reads
   against the keys the factories actually produce. It reports 0.

6. CANVASITEM API ON A NODE SCRIPT.
   _graph_center() used get_viewport_rect(), which is CanvasItem-only,
   in KeystoneViewer.gd -- a script that extends Node. The earlier
   tooltip fix worked only because it was called on tooltip_panel (a
   Control), not on self.
   LESSON: check the base class before using node-type-specific API.
   GDScript only surfaces this at parse time.

7. A CUT SYSTEM LEAVES READERS BEHIND.
   After the crafting rip-out, KeystoneViewer._append_cert_and_recipe_lines
   still read GameData.recipes and threw at runtime every time the talent
   panel opened. It had been repointed from "Crafting" to "Auxiliary"
   during the keystone consolidation -- correct then, a landmine once the
   array was deleted.
   LESSON: when a system is cut, grep for EVERY reader of its data,
   including UI files that are not in the working set at the time.
