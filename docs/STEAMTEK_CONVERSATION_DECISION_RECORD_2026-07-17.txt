OPENING CONTEXT AND CORRECTION

The session began with a detailed carryover summary describing Steamtek as a compact-surface, deep-silo, neo-industrial cyberpunk-steampunk isometric RPG. The summary described the art target as a high-fidelity 2.5D hybrid using modular painted/isometric environments, 3D characters and effects, and a locked 2:1-style camera. It also said the project had moved toward a stricter modular_v1 snap-system standard.

The user immediately corrected that statement: modular_v1 is not the standard. This correction governs all later work. Older modular experiments may exist in the repository, but current character, camera, dimensional environment, material, rain, and steam decisions must not be presented as modular_v1 work. Reusable effects are allowed, but reusability is not the same as adopting a modular building standard.

RIG, ANIMATION, AND SHARED-SKELETON DISCUSSION

The user asked whether animation would be done in Meshy. The decision was yes for fast animation sourcing, while Blender remains the fallback for cleanup, rig inspection, sockets, scale calibration, LOD work, or animation repair. The user asked whether a wire mesh was required. The clarification was that a visible wireframe is not a game requirement; what matters is a skinned polygon mesh bound to a skeleton.

The user asked whether future characters and enemies with different appearances would use the same rig. The direction is to use a shared humanoid skeleton when body proportions are compatible because it permits animation reuse. Distinct creatures or strongly different silhouettes may require their own skeletons. Sharing a humanoid rig does not mean every character must look the same.

In Meshy, the protagonist was rigged and animation actions were added. The user encountered repeated “Fail to add animation action” notifications for some attempts. Later screenshots showed three successfully added actions: Idle 3, Running, and Walking. The user asked whether to include running; the answer was yes, because the controller would need walk and run states. The download settings were configured for GLB with all added animations so the rig and clips remained together.

The user then placed all three results into the intake workflow and supplied videos of the animations. A clean production package was created with:

A screenshot of the legacy apartment proof showed the character next to an attractively lit but dimensionally unreliable painted building. The session then moved to a measured dimensional apartment and alley blockout. Early screenshots exposed a roof/side-access problem where the character could walk beneath or behind an unmodeled portion of the shell. A full invisible building-footprint collision blocker was added to prevent entry from unmodeled sides and stop roof clipping.

The user asked whether the measured building and alley were based on the chosen project vision and scale, and whether the scale matched other 2.5D isometric games. The answer was that the selected measurements are within credible game scale and work with the chosen locked camera. The user explicitly said to lock the system in.

The resulting live-3D scale contract is:
- 1 Godot unit equals 1 meter
- C001 stays at scale 1
- Door is 1.2 meters wide by 2.2 meters high
- Storeys are 3.2 meters
- Two-storey facade is 6.4 meters
- Reference apartment footprint is 10 by 7 meters
- Service alley is 3.5 meters
- Documented gameplay-clearance widening may use 10–20 percent
- Environment fits the character; the character is never resized to old artwork

The camera contract is:
- Orthographic projection
- 60-degree azimuth
- 30-degree elevation
- Zero roll
- Reference camera position Vector3(8.66, 6.77, 5)
- Orthographic size 8.5 for close exploration

The user repeatedly tested the blockout, approved the scale and camera, and asked to continue.

HAND-PAINTED ART DIRECTION IN THE LIVE-3D SHELL

The user asked whether the environment work was based around the selected hand-painted asset style. The decision was yes: measured 3D geometry provides scale, collision, and lighting, while painterly textures and visual layers provide the final look. This is a hybrid rather than a choice between entirely painted 2D or entirely realistic 3D.

An initial facade texture was generated but looked too much like a material scan. It was retained only as a fallback. A second version was generated with stronger hand-painted broad brushwork and neutral lighting. It became:
STK_MAT_ApartmentFacade_HandPainted_v02.png

It was applied through world-triplanar mapping. Separate geometry supplied window frames, door trim, base courses, facade pipe and clamps, roof steel, roof vent, amber door header light, and access panel. Live cyan, magenta, and amber light energy was reduced because the earlier result was too flat and neon-heavy. The later screenshot showed readable wall brushwork, better trim, and improved character lighting. The facade passed the first material gate, although windows remained placeholders.

The next ground pass generated:
STK_MAT_WetConcrete_HandPainted_v01.png

The prompt required a seamless top-down painterly dark blue-gray industrial concrete albedo with broad brushwork, subtle cracks, patched areas, water staining, aggregate, muted copper grime, neutral lighting, and no baked neon, reflections, objects, text, Victorian ornament, or photographic scan look. The texture was copied into the project and assigned to the sidewalk, world ground, and alley via triplanar mapping. Separate material settings give those zones different wetness/readability.

Puddles were created as shallow, non-colliding oval meshes with a glossy material. This deliberately keeps reflection control separate from the texture. An industrial drainage channel and grate were added to the alley. The user said the result was good.


RAIN AND STEAM DECISION

The user asked whether rain and steam are created as assets or through Godot scenes. The answer was mostly through Godot scenes: GPUParticles3D for rain and steam, with reusable vent/pipe geometry and optional small texture resources. This avoids hand-painted animation frames and permits later performance tuning.

The user asked to proceed with rain and steam. Existing repository rain, splash, mist, and steam files were inspected. Those systems were legacy Node2D effects and were not appropriate for the approved live-3D path. New standalone packed 3D effect scenes were therefore created:

res://scenes/effects/live3d/SteamtekRainVolume3D.tscn
res://scenes/effects/live3d/SteamtekSteamVent3D.tscn

During this work, the user asked whether any of it was “our modular tech.” The answer was explicitly no. The effects are reusable packed Godot scenes, but they do not rely on modular_v1, modular snap standards, modular geometry, or old modular tech. They are independent environmental effects that can be instanced in any live-3D level. The approved camera and live-3D scale contract remain authoritative.

The rain uses GPUParticles3D streaks in a localized volume under the CameraRig. Because the rig follows the character, rain coverage moves with the camera instead of requiring a huge always-active emitter. The steam uses translucent billboarded puffs with gradual growth and fade. It is placed at the roof vent and a new alley steam pipe.

The new files, references, and text formatting passed static checks. Godot was not launched through the command line because of the recurring crash history. At the close of the recorded work, the rain and steam still required normal-editor F6 visual confirmation from the user.
